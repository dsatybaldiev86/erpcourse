Why Scale? The Problem with Single Servers
0 – 5 min
Instructor note

Open with a real scenario: "Imagine your app goes viral overnight — 10,000 users hit it at once. What happens?" Let students guess. Then build the case for why a single server always fails at scale.

The Single Server Problem

Core Problem

A single server is the simplest architecture — one machine handles all requests, holds all data, and runs all logic. It works perfectly during development, but fails in three critical ways as systems grow.

Failure 1 — Downtime

Any hardware failure, crash, or deployment takes the entire system offline. No redundancy = no recovery. Every minute of downtime is lost revenue and user trust.

SPOF — Single Point of Failure
Failure 2 — Slow Response

As traffic grows, the server becomes overloaded. CPU, memory, and I/O are shared across all requests. One slow query can stall everyone. Response time increases non-linearly.

Bad UX → User Churn
Failure 3 — Growth → System Collapse

Vertical scaling (buying a bigger server) has a hard ceiling. You cannot add RAM or CPU indefinitely. Eventually, no single machine can serve 10M users, and the system needs to be redesigned from scratch — which is far more expensive than planning for scale early.

Vertical Scaling has limits Horizontal Scaling is the answer

Three Core Goals We'll Solve Today

Goal 1
Scalability
Handle more users by adding more machines horizontally, not just upgrading one.
Goal 2
High Availability
Stay operational even when individual components fail. No single point of failure.
Goal 3
Performance
Respond fast regardless of user location or system load through caching and distribution.

Load Balancer & API Gateway
5 – 12 min

Load Balancer

What it does

A Load Balancer sits between clients and your server pool. It receives incoming requests and distributes them across multiple backend servers using a defined algorithm. The client never knows which server it's talking to.

Think of it like a traffic controller at an intersection — it keeps traffic flowing evenly so no single road becomes a bottleneck.

Why we need it

  • Prevents overload — distributes requests so no single server is overwhelmed
  • Improves availability — if one server fails, others continue serving requests
  • Enables horizontal scaling — add more servers and the LB uses them automatically
  • Health checks — regularly pings servers and removes unhealthy ones from rotation
Basic Request Flow
Client
→
Load Balancer
distributes to ↓
Server A
Server B
Server C
All servers run identical application code

Balancing Algorithms — Choose the right one

Algorithm 1
Round Robin
Requests are sent to servers in strict rotation: A → B → C → A → B → C. Simple, works well when all servers have equal capacity.
Algorithm 2
Least Connections
Sends each new request to the server with the fewest active connections. Better for variable-length requests (e.g. file uploads vs. simple API calls).
Algorithm 3
IP Hash
Hashes the client's IP address to determine which server handles their requests. Ensures a user always hits the same server — useful for session-based apps.
When to use which algorithm
  • Round Robin — Stateless microservices where requests are equal in cost
  • Least Connections — Media streaming, file processing with variable request duration
  • IP Hash — Shopping cart or session-based apps that require server affinity

API Gateway

What it does

An API Gateway is a single entry point that all client requests flow through before they reach your services. It handles cross-cutting concerns — things every service needs — in one place rather than duplicating logic across dozens of microservices.

If a Load Balancer is the traffic controller, the API Gateway is the security checkpoint and reception desk — it verifies who you are, decides where you go, and logs everything.

Responsibilities

  • Authentication — validates JWT tokens, API keys, or OAuth before requests reach services
  • Routing — maps /api/orders → Orders Service, /api/users → Users Service
  • Rate Limiting — blocks clients that exceed request quotas (e.g. 1000 req/min)
  • Logging & Monitoring — centralized request logging for observability
  • SSL Termination — handles HTTPS at the edge; internal traffic uses HTTP
  • Request Transformation — converts legacy XML to JSON, merges responses
Full Request Stack
Client (Browser / App)
↓
API Gateway control layer
↓
Load Balancer distribution layer
↓
Orders
Users
Payments
Key distinction to emphasize

Gateway ≠ Load Balancer. The API Gateway is a smart, application-aware layer that handles business-level concerns (auth, routing, rate limiting). The Load Balancer is a network-level layer that distributes traffic by connection/IP. They solve different problems and work together — gateway sits upstream of the load balancer.

Single Point of Failure Warning

The API Gateway can itself become a bottleneck or SPOF. Production systems always deploy multiple gateway instances behind a load balancer to ensure the gateway is itself highly available.

Databases & Scaling Strategies
12 – 20 min
The Database Bottleneck Problem

Application servers are easy to scale — just add more of them behind a load balancer. But databases are stateful: they hold your data, so you can't simply spin up ten copies. A single database becomes the chokepoint of the entire system under heavy read or write load. Two strategies address this: Replication (copying data) and Sharding (splitting data).

Database Replication

Core Concept

Replication copies data from a primary (master) database to one or more replica (slave) databases. Writes always go to the primary; reads can be distributed across replicas. The primary asynchronously pushes changes to replicas using a write-ahead log.

Master–Slave (Primary–Replica)

Primary DB (writes)
↓ replicates to
Replica 1 (reads)
Replica 2 (reads)
  • Application writes go to Primary only
  • Read queries are spread across Replicas
  • If Primary fails → promote a Replica

Multi-Master Replication

Master A
⇄
Master B
Both accept reads & writes; sync each other
  • Both nodes accept writes
  • Useful for geographic distribution
  • Requires conflict resolution logic
Benefits
  • High Availability — replicas serve traffic if primary fails
  • Read Scaling — 10 replicas = 10× read throughput
  • Backup — replicas can serve as live backups
  • Low latency reads — replicas can be placed closer to users
Trade-offs
  • Eventual consistency — replicas may lag behind primary by milliseconds
  • Read-your-writes problem — user may not immediately see their own write
  • Replication lag — under heavy write load, lag can grow
  • Write bottleneck remains — all writes still go to one primary
Real-World Use

Instagram uses read replicas extensively. Their user feed reads outnumber writes by ~100:1. By routing reads to replicas, the primary only handles new post writes, dramatically reducing its load.

Sharding — Horizontal Database Scaling

Core Concept

Sharding splits your data horizontally across multiple databases called shards. Each shard holds a distinct subset of the full dataset. Unlike replication (where all shards have all data), each shard has only its portion of the data. A shard key determines which shard stores each record.

Without Sharding

1 DB → 10 million users
↓

Overloaded. Slow. Single point of failure.

With Sharding

DB Shard 1 → users 1 – 1,000,000
DB Shard 2 → users 1M – 2,000,000
DB Shard 3 → users 2M – 3,000,000

Hash Sharding Strategy

How it works

Apply a hash function to the shard key (e.g. user_id), then use the modulo operator to determine which shard receives the record. This distributes data evenly across shards regardless of the data's natural distribution.

Hash Sharding Example — Insert user_id = 123, 3 shards total
user_id = 123
→
hash(123) = 456789
→
456789 % 3 = 0
→
Shard 0
user_id 124: hash(124) % 3 = 1 → Shard 1  |  user_id 125: hash(125) % 3 = 2 → Shard 2
Benefits
  • Even distribution — hash function prevents data concentration
  • Avoids hotspots — no single shard becomes overloaded
  • Write scalability — writes are distributed across all shards
Trade-off — Rebalancing Problem
  • Adding a 4th shard changes the modulo: % 4 instead of % 3
  • Almost all records now map to a different shard
  • Requires migrating the majority of data — expensive and risky
  • Solution: Consistent Hashing minimizes reshuffling (advanced topic)
Replication vs. Sharding — When to use each
  • Replication → when reads far outnumber writes; need high availability
  • Sharding → when data volume or write load exceeds what one machine can handle
  • Both together → production-grade systems use both: each shard is also replicated

High Availability Architecture
20 – 30 min
Definition

High Availability (HA) means a system stays operational and accessible even when individual components fail. HA is measured as a percentage of uptime over time. The goal is to eliminate every single point of failure so there is no one component whose failure takes down the system.

99%
~3.6 days downtime/year
99.9%
~8.7 hours/year
99.99%
~52 minutes/year
99.999%
~5 minutes/year ("Five 9s")

Technique 1 — Redundancy

Concept

Redundancy means running multiple identical instances of every critical component. If one fails, the others continue serving requests without interruption. There is no single component that, if removed, breaks the system.

  • Multiple application servers behind a load balancer — any one can be taken down
  • Multiple DB replicas — reads continue even if two replicas fail simultaneously
  • Multiple load balancers — active-passive or active-active pairs
  • Multiple API Gateway instances — always deploy more than one
  • Multiple availability zones — AWS/GCP zones within a region are physically isolated

Technique 2 — Failover

Concept

Failover is the automatic process of switching to a backup component when the active one fails. The switchover should be fast enough (ideally milliseconds) that users experience at most a brief pause, not an outage.

Active–Passive Failover

One active node handles all traffic. A passive standby node is ready but idle. On failure, the passive node is promoted and traffic is redirected to it.

SimplerSome recovery time
Active–Active Failover

Both nodes handle traffic simultaneously. If one fails, the other absorbs all traffic with no downtime. Also provides load distribution.

Technique 3 — Multi-Region Deployment

Concept

Multi-region deployment runs your entire system stack in multiple geographic locations simultaneously. Each region is independent — it has its own application servers, databases, and supporting infrastructure. A Global Load Balancer routes each user to the closest region.

This protects against regional disasters (data center fire, natural disaster, network outage) and also dramatically reduces latency for geographically distributed users.

Benefits
  • Disaster recovery — entire region can go down without global outage
  • Low latency — users connect to the nearest region
  • Data sovereignty — EU user data stays in EU for compliance
  • Traffic isolation — a spike in one region doesn't affect others
Challenges
  • Data consistency — syncing databases across regions introduces lag
  • Cost — running full stacks in 3 regions triples infrastructure cost
  • Conflict resolution — concurrent writes to different regions need careful handling
  • Operational complexity — deployments must be coordinated across all regions
Real World — Netflix Multi-Region

Netflix operates in 3 AWS regions simultaneously (US East, US West, EU). When the entire US East region had a major outage in 2012, Netflix automatically shifted all traffic to US West with no user-facing downtime. Their multi-region investment paid for itself in that one incident.

Performance Optimization — CDN
30 – 35 min

Content Delivery Network (CDN)

What it does

A CDN is a globally distributed network of edge servers — also called Points of Presence (PoPs) — that cache content close to users. When a user requests an asset, the CDN serves it from the nearest edge server instead of sending the request all the way to your origin server.

The fundamental insight: the speed of light is a real constraint. A request from Bishkek, Kyrgyzstan to a server in Virginia, USA travels ~10,000 km. A CDN edge server in Frankfurt is ~4,000 km closer. Physics alone makes CDN-served content faster.

Without CDN vs. With CDN — User in Bishkek, Kyrgyzstan loading an image

Without CDN

User (KG)
→
Origin (US)
~250ms round-trip latency

With CDN

User (KG)
→
Edge (Frankfurt)
~40ms round-trip latency (~6× faster)

What to cache on CDN

  • Images & videos — the majority of page weight
  • Static files — CSS, JavaScript, fonts, HTML templates
  • API responses — non-personalized, read-heavy endpoints
  • Software downloads — large, infrequently changing files
Never cache: user-specific data Never cache: real-time data

Benefits

  • Faster load times — content served from nearby edge
  • Reduced backend load — origin receives only cache misses
  • DDoS protection — CDN absorbs attack traffic at the edge
  • Better SEO — page speed directly impacts search ranking
  • Cost savings — less bandwidth consumed at origin
How CDN Cache Works

Cache Hit: User requests image.jpg. Edge server has it cached → serves it immediately, no origin request.

Cache Miss: User requests new-image.jpg. Edge server doesn't have it → fetches from origin, caches it, serves it. Subsequent users get the cached version.

Cache expiry is controlled via HTTP headers: Cache-Control: max-age=86400 tells CDN to cache for 24 hours. After expiry, the CDN re-fetches from origin.

CDN Providers & Their Reach
  • Cloudflare — 300+ edge locations worldwide, includes DDoS protection
  • AWS CloudFront — integrated with AWS infrastructure, 450+ PoPs
  • Akamai — oldest, largest; 4000+ PoPs, used by major enterprises and governments
  • Fastly — real-time purging, popular for dynamic content caching
Discussion prompt

"Where else could we add caching besides the CDN edge?" — Expected answers: in-memory cache (Redis/Memcached) inside the application tier, database query cache, browser cache, reverse proxy cache (Nginx). Let students explore the idea that caching can exist at multiple levels of the stack.

Discussion Questions

Use these to close the lecture and check understanding. Give students 1 minute to think before taking answers.

  • 1
    When should we use sharding instead of replication? — Think about what problem each solves. Replication addresses availability and read scaling. Sharding addresses data volume and write scaling. When your total data size or write throughput exceeds what one server can handle, sharding becomes necessary.
  • 2
    What happens if the API Gateway itself fails? — The entire system becomes inaccessible, because all traffic routes through it. This means the API Gateway must itself be highly available: run multiple instances, behind a load balancer, in multiple availability zones. The gateway cannot be a single point of failure.
  • 3
    Why is eventual consistency acceptable in some systems? — A replica being 50ms behind the primary is fine for a social media feed — seeing a post appear slightly later doesn't matter. But for a bank balance or inventory count, strong consistency is required. The CAP theorem formalizes this trade-off: in a distributed system, you must choose between Consistency and Availability when a network partition occurs.
  • 4
    Where else would you add caching besides the CDN? — Expected answers: in-memory cache (Redis/Memcached) between app servers and database; browser cache via HTTP headers; application-level memoization; database query cache. Every layer of the stack can benefit from caching appropriately scoped data.

Lecture Summary

Scalable, highly available systems are built in layers. No single technique solves everything. A Load Balancer distributes traffic, but breaks without redundancy. Replication improves reads and availability, but writes still bottleneck on one primary. Sharding solves data scale, but complicates rebalancing. Multi-region solves geographic resilience, but introduces consistency challenges. CDN improves performance for static content, but not for dynamic data.

The lesson: every architectural decision trades one problem for another. A senior engineer's skill is knowing which trade-off is acceptable for their specific context — user base, data model, team size, and cost constraints.
